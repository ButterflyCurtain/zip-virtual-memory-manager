  SECTION 1 : Diff Layer  —  Dirty Page Lifecycle
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Dirty pages are never silently discarded.  They are held in one of two
  tiers at all times until commit() or an explicit discard() call.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Diff Layer — Tier 1 : Hot Dirty Store  ( in-memory )                   │
  │                                                                          │
  │   dirty_limit   = 256 MB  ( default; configurable per mount )            │
  │   dirty_current = Σ sizeof(modified pages currently in Tier 1)          │
  │                                                                          │
  │   [ pg: file_a:0  ] dirty  4 KB                                         │
  │   [ pg: file_b:12 ] dirty  4 KB                                         │
  │   [ pg: file_c:0  ] dirty  4 KB    dirty_current = N × 4 KB             │
  │   [ ...           ] ...                                                  │
  │                                                                          │
  │   write(path, offset, data)                                              │
  │        │                                                                 │
  │        ├─ dirty_current + page_size ≤ dirty_limit                        │
  │        │       └─→ store in Tier 1, return immediately                   │
  │        │                                                                 │
  │        └─ dirty_current + page_size  >  dirty_limit   ← PRESSURE POINT  │
  │                └─→ trigger Spill  ( see Section 1.1 for spill modes )    │
  └──────────────────────────────────┬──────────────────────────────────────┘
                                     │  spill triggered
                                     ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Diff Layer — Tier 2 : Spill File  ( on-disk )                          │
  │                                                                          │
  │   path :   archive.zip.vmm/vmdirty  ( alongside vmidx )                    │
  │   format : append-only journal of DATA RECORDs and COMMIT MARKERs       │
  │            ( see vmdirty Journal Specification for binary layout )       │
  │                                                                          │
  │   Spill policy : oldest-written pages move to Tier 2 first              │
  │                  ( FIFO — preserves most-recently-written pages hot )    │
  │                                                                          │
  │   Read path priority:                                                    │
  │     Tier 1 (hot)  →  Tier 2 (spill)  →  Page Cache  →  Seek Index      │
  │                                                                          │
  │   On process crash : vmdirty survives on disk.                          │
  │   On next open     : VMM detects vmdirty, invokes recovery protocol.   │
  └──────────────────────────────────────────────────────────────────────────┘

  ┌───────────────────────────────────────────────────────────────────────┐
  │  dirty_limit configuration                                            │
  │                                                                       │
  │    mount(path)                        default (256 MB)               │
  │    mount(path, dirty_limit=N)         explicit limit in bytes         │
  │    mount(path, dirty_limit=0)         spill-only (no Tier 1 budget)  │
  │    mount(path, dirty_limit=UNLIMITED) no spill; all dirty pages      │
  │                                       remain in Tier 1 until         │
  │                                       commit() or discard()          │
  └───────────────────────────────────────────────────────────────────────┘

  dirty_limit=UNLIMITED disables the spill mechanism entirely.  All dirty
  pages remain resident in Tier 1 for the duration of the session.  Memory
  consumption grows proportionally to the total volume of unique modified
  pages.  No vmdirty file is written.  The application is responsible for
  ensuring sufficient address space is available for the expected dirty
  page volume.

  Invariants:

    I-1  A page in Tier 1 or Tier 2 is removed only by:
           (a) commit()             — page written to new ZIP
           (b) discard()           — explicit caller decision
           (c) recover_committed() — recovery decision
           (d) recover_all()       — recovery decision
         A dirty page is never removed by LRU eviction, OOM killer,
         or any implicit path.

    I-2  dirty_current always equals the sum of page_size for all pages
         currently resident in Tier 1.  Pages in Tier 2 do not count
         toward dirty_current.

    I-3  The in-memory Tier 2 index is always consistent with vmdirty
         on disk.  If a record was appended to disk, it is reflected in
         the index (or superseded by a newer record for the same page).

  SECTION 1.1 : Spill Execution Modes
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Two spill execution modes govern whether the disk I/O associated with a
  spill operation blocks the calling write() or runs in the background.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Synchronous Spill  ( --sync-spill, default )                           │
  │                                                                          │
  │   Spill executes inline within the write() call that triggered it.      │
  │   The caller blocks until enough pages have been written to vmdirty    │
  │   to bring dirty_current below dirty_limit.                             │
  │                                                                          │
  │   Suitable for:  single-threaded callers, batch workloads where         │
  │                  write latency is not latency-sensitive.                 │
  └──────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Asynchronous Spill  ( --async-spill )                                  │
  │                                                                          │
  │   A dedicated spill thread drains Tier 1 pages to vmdirty in the      │
  │   background.  The write() call that crosses dirty_limit returns to     │
  │   the caller immediately after enqueueing the incoming page.            │
  │                                                                          │
  │   dirty_high_watermark                                                   │
  │       = dirty_limit × spill_headroom_factor  ( default factor: 1.5 )   │
  │                                                                          │
  │   Backpressure:  if dirty_current reaches dirty_high_watermark,         │
  │       write() blocks until the spill thread drains Tier 1 back below    │
  │       dirty_limit.  This bounds worst-case memory use while keeping     │
  │       the common case non-blocking.                                     │
  │                                                                          │
  │   Thread safety:                                                         │
  │       Tier 1 deque and VmdirtyWriter share a single mutex.              │
  │       write() holds the mutex only for the deque insertion step;        │
  │       the disk write is performed by the spill thread outside the       │
  │       mutex.  Reads to Tier 1 and Tier 2 proceed concurrently with      │
  │       spill via a reader-writer lock.                                   │
  │                                                                          │
  │   flush() and commit() in async mode:                                   │
  │       Both operations signal the spill thread to drain completely,      │
  │       then wait for the drain to finish before proceeding.              │
  │       This preserves the durability guarantee of flush() (STRICT).     │
  │                                                                          │
  │   Suitable for:  interactive or latency-sensitive workloads where       │
  │                  write spike latency must be bounded.                   │
  └──────────────────────────────────────────────────────────────────────────┘

  Mount option summary:

    mount(path)                              sync spill, dirty_limit=256 MB
    mount(path, --async-spill)               async spill, headroom factor=1.5
    mount(path, --async-spill,
               spill_headroom_factor=2.0)   async spill, custom headroom

  The spill execution mode is orthogonal to the fsync mode (--sync-spill /
  --lazy-spill) and the spill victim-selection policy (FIFO / LRU / CLOCK).
  Any combination of these three axes is valid.

  ┌───────────────────────────────────────────────────────────────────────────┐
  │  Async spill — state diagram                                              │
  │                                                                           │
  │   write()                     Spill thread                               │
  │   ───────                     ────────────                               │
  │   enqueue page                                                            │
  │   dirty_current ≤ dirty_limit → return immediately                        │
  │   dirty_limit < dirty_current < dirty_high_watermark → return immediately │
  │   dirty_current ≥ dirty_high_watermark → block on condition variable      │
  │                               dequeue victim                              │
  │                               write DATA RECORD to vmdirty              │
  │                               dirty_current -= page_size                  │
  │                               if dirty_current < dirty_limit:             │
  │                                   signal condition variable               │
  │                               (blocked write() wakes and returns)         │
  └───────────────────────────────────────────────────────────────────────────┘

  SECTION 2 : External Modification Detection
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Single-metric detection (mtime alone, or inode alone) is unreliable across
  filesystems and network mounts.  The VMM uses a layered fingerprint with
  cost-proportional fallback.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Fingerprint  —  stored in vmidx header at index-build time            │
  │                                                                          │
  │   field                source                    cost to check           │
  │   ───────────────────────────────────────────────────────────────────── │
  │   file_size            stat().st_size            O(1)                   │
  │   inode                stat().st_ino             O(1)                   │
  │   cd_hash              XXH3-128 of Central       O(cd_size) ← fast      │
  │                        Directory block only                              │
  │   mtime_ns             stat().st_mtime_ns        O(1), runtime ESTALE    │
  │                                                  trigger + diagnostics   │
  │   full_hash            XXH3-128 of entire file   O(file_size) ← slow    │
  │                        ( computed only in strict-fingerprint mode )      │
  └──────────────────────────────────────────────────────────────────────────┘

  Validation cascade on open():

  open( archive.zip )
       │
       ├─[1] FAST CHECK  — O(1)
       │       Compare: file_size  AND  inode
       │       Match   → proceed to step 2
       │       Mismatch→ ──────────────────────────────────────→ INVALIDATE
       │
       ├─[2] MEDIUM CHECK  — O(cd_size), typically < 1 ms
       │       Compare: cd_hash  ( Central Directory only )
       │       Match   → index valid, mount immediately  ✓
       │       Mismatch→ ──────────────────────────────────────→ INVALIDATE
       │
       └─[3]  mtime_ns is recorded in the vmidx header.  It is not used
              as a validity gate at open() — it is unreliable on NFS, FAT,
              and some distributed filesystems, and cd_hash is checked at
              open() regardless — but it participates as a change TRIGGER
              in the runtime ESTALE check, where it catches same-size
              in-place modification that size+inode cannot (see main
              architecture document, SNAPSHOT CONSISTENCY).

  Hash algorithm note:

    cd_hash and full_hash are integrity checks against accidental
    corruption and out-of-band modification.  They are not message
    authentication codes and prove nothing about origin; authenticity
    is out of scope (see UNTRUSTED ARCHIVES in the main architecture
    document).  XXH3-128 is chosen for throughput (an order of
    magnitude faster than cryptographic hashes, relevant for
    --strict-fingerprint on multi-GB archives) with a 128-bit state
    space that makes accidental collision negligible.

  INVALIDATE path:
       │
       ├─ vmdirty present?
       │       YES → surface CONFLICT to caller:
       │               source ZIP changed while dirty pages exist.
       │               caller must resolve:  discard_dirty() | abort()
       │               VMM does not auto-resolve.
       │       NO  → silently rebuild vmidx from scratch
       │
       └─ rebuild proceeds as first-open initialization


  Strict fingerprint mode  ( mount option: --strict-fingerprint )

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  --strict-fingerprint                                                    │
  │                                                                          │
  │  Adds a full XXH3-128 of archive.zip at every open(), regardless of     │
  │  steps [1] and [2].  Correct even when mtime is stale or inode is       │
  │  recycled.                                                               │
  │                                                                          │
  │  Intended for: NFS mounts, object storage with FUSE bridge,             │
  │                shared caches across multiple hosts.                      │
  └─────────────────────────────────────────────────────────────────────────┘

  SECTION 3 : File Lifecycle State Machine
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

                        open()
                          │
                          ▼
                   ┌─────────────┐
                   │   CHECKING  │  fingerprint validation cascade
                   └──────┬──────┘
               valid /         \ invalid
                    /           \
                   ▼             ▼
          ┌──────────────┐   ┌─────────────────────┐
          │    CLEAN     │   │  vmdirty present?  │
          │  (mounted,   │   └────────┬────────────┘
          │  no changes) │       yes  │  no
          └──────┬───────┘           │   │
                 │ write()           │   ▼
                 ▼                   │  ┌──────────────┐
          ┌──────────────┐           │  │  REBUILDING  │──→ CLEAN
          │    DIRTY     │           │  └──────────────┘
          │  Tier1/Tier2 │           ▼
          └──────┬───────┘     ┌────────────┐
          within │ limit       │  CONFLICT  │
          ───────┤             │  caller    │
          exceed │             │  resolves  │
                 ▼             └────────────┘
          ┌──────────────┐
          │   SPILLING   │  Tier 1 → Tier 2
          │              │  ( sync: inline with write() )
          │              │  ( async: background thread  )
          └──────┬───────┘
                 │ spill complete
                 ▼
          ┌──────────────┐
          │    DIRTY     │  (continues accepting writes)
          └──────┬───────┘
                 │ commit()
                 ▼
          ┌──────────────┐
          │  COMMITTING  │  recompress → write .new.zip → rename()
          └──────┬───────┘
                 │ rename() done
                 ▼
               CLEAN

  State definitions:

    CHECKING    Fingerprint validation is in progress.
    CLEAN       Mounted with no uncommitted modifications.
    DIRTY       One or more pages are modified and uncommitted.
                Sub-state: pages may be in Tier 1, Tier 2, or both.
                With dirty_limit=UNLIMITED: all dirty pages remain in
                Tier 1; Tier 2 is never used.
    SPILLING    A spill operation is moving pages from Tier 1 to Tier 2.
                In synchronous mode: write() is blocked until spill completes.
                In asynchronous mode: spill thread is active; write() returns
                immediately unless dirty_high_watermark is reached.
                Reads and writes are accepted in both modes.
                Not reachable when dirty_limit=UNLIMITED.
    REBUILDING  vmidx is being reconstructed from the source ZIP.
    COMMITTING  commit() is in progress.
    CONFLICT    Source ZIP changed while vmdirty dirty pages exist.
                The VMM is suspended; caller must resolve before mount
                can proceed.
